풀스택 프로젝트 진행하는데 기능정의서 사용하는데 스네이크로 표기 하기로 했거든? 전체적으로 변수를 다 스네이크로 표기하는거야?

핵심 정리 한 줄

👉 문서·DB·API 기준은 스네이크,
👉 프론트 JS/React 변수는 카멜,
👉 언어 컨벤션은 언어 따라간다

1️⃣ 기능정의서 / 요구사항 문서 ✅ (스네이크 추천)

사람 + 개발자 + 기획자 모두 보는 문서

DB 필드랑 1:1 매칭 목적

user_id
user_name
created_at
is_deleted

👉 기능정의서에서는 전부 스네이크로 통일하는 게 좋아
(나중에 DB 설계, API 명세랑 연결 쉬움)

2️⃣ DB 컬럼명 ✅ (스네이크 거의 국룰)
user_id
email
created_at
updated_at
del_yn


MySQL / MariaDB / PostgreSQL 전부 스네이크가 표준에 가까움

대소문자 이슈도 없음

👉 DB는 무조건 스네이크 추천

3️⃣ API(JSON) ⚠️ 팀 컨벤션에 따라

둘 중 하나로 딱 정해야 함

✅ 백엔드가 Python / FastAPI 계열이면
{
  "user_id": 1,
  "user_name": "현서",
  "created_at": "2026-01-23"
}

⚠️ 프론트가 React 중심이면
{
  "userId": 1,
  "userName": "현서",
  "createdAt": "2026-01-23"
}


👉 이 경우 백엔드 ↔ 프론트 변환 계층 둠
(serializer / mapper)

4️⃣ 프론트엔드 (React / JS) ❌ 스네이크 비추

JS 세계에서는 카멜 케이스가 표준

const userId = 1;
const userName = "현서";
const createdAt = new Date();


❌ 이렇게 쓰면 어색함

const user_id = 1; // JS에선 안 예쁨

5️⃣ 백엔드 코드 (Python 기준)

Python은 스네이크 케이스가 공식 컨벤션(PEP8)

def get_user_info(user_id):
    created_at = datetime.now()


👉 여기서는 스네이크 OK

결론!!!!
실무에서 가장 깔끔한 패턴 (추천 ⭐)
[기능정의서]  user_id
      ↓
[DB]          user_id
      ↓
[Backend]     user_id (Python)
      ↓ 변환
[Frontend]    userId (React)

기능정의서에 이렇게 한 줄 써두면 끝남

!!!!!네이밍 규칙

기능정의서 / DB / Backend : snake_case

Frontend(JavaScript) : camelCase

API 응답은 Frontend 기준으로 camelCase 사용
