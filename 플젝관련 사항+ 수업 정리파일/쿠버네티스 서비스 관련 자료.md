발표 주제
Kubernetes Service 완전 이해
1️⃣ 오프닝 (1분)
🖼 그림 1 — 문제 상황
[ Frontend Pod ]  --->  [ Backend Pod 1 ] (10.244.1.3)
                      [ Backend Pod 2 ] (10.244.2.7)
🎙 대본

Kubernetes에서 Pod는 고정 IP가 아닙니다.
재시작되거나 재스케줄링되면 IP가 바뀝니다.

그렇다면 Pod IP로 직접 통신하면 어떤 문제가 생길까요?

네, 장애가 발생합니다.

이 문제를 해결하기 위해 등장한 것이 Service입니다.

2️⃣ Service 개념 (2분)
🖼 그림 2 — Service 등장
[ Frontend ]
     ↓
[ backend-service ]
     ↓
[ Backend Pods ]
🎙 대본

Service는 Pod들을 라벨 기반으로 묶고
고정된 IP와 DNS 이름을 제공합니다.

그리고 내부적으로 로드밸런싱을 수행합니다.

즉, “Pod 앞에 붙는 네트워크 추상화 계층”입니다.

3️⃣ Service는 어디와 연계되는가? (핵심 포인트)

Service는 단독으로 동작하지 않습니다.

🖼 그림 3 — 내부 동작
Client
   ↓
Service (ClusterIP)
   ↓
kube-proxy
   ↓
Pod
🎙 대본

Service가 트래픽을 직접 전달하는 것이 아닙니다.
실제 전달은 kube-proxy가 담당합니다.

kube-proxy는 iptables 또는 ipvs 규칙을 생성해
트래픽을 Pod로 전달합니다.

DNS 이름은 CoreDNS가 관리합니다.

Pod 간 네트워크는 CNI 플러그인이 담당합니다.
예를 들어
Calico
Flannel
같은 것들이 있습니다.

4️⃣ Service 종류 (5~6분 핵심)
4-1️⃣ ClusterIP
🖼 그림
[ React Pod ]
      ↓
[ ClusterIP Service ]
      ↓
[ Backend Pods ]
🎙 대본

기본 타입입니다.
클러스터 내부에서만 접근 가능합니다.

마이크로서비스 구조에서
서비스 간 통신에 사용됩니다.

4-2️⃣ NodePort
🖼 그림
외부
   ↓
NodeIP:30000
   ↓
NodePort
   ↓
ClusterIP
   ↓
Pod
🎙 대본

NodePort는 외부에서 노드 IP와 특정 포트로 접근합니다.

내부적으로는 ClusterIP를 포함합니다.

개발 환경이나 테스트 환경에서 주로 사용됩니다.

예시 YAML:

apiVersion: v1
kind: Service
metadata:
  name: app-sv
spec:
  type: NodePort
  selector:
    app: app-label
  ports:
    - port: 80
      targetPort: 80
      nodePort: 30000
4-3️⃣ LoadBalancer

클러스터가
Amazon Web Services
같은 클라우드에 있다면:

🖼 그림
Internet
   ↓
Cloud LoadBalancer
   ↓
NodePort
   ↓
Pod
🎙 대본

LoadBalancer 타입을 생성하면
클라우드의 로드밸런서가 자동 생성됩니다.

실제 운영 환경에서 가장 많이 사용됩니다.

4-4️⃣ ExternalName
🖼 그림
Pod
  ↓
ExternalName Service
  ↓
외부 DNS
🎙 대본

외부 서비스와 DNS 기반으로 연결합니다.
Pod를 직접 선택하지 않습니다.

5️⃣ 포트 개념 정리 (중요)
🖼 그림
외부(30000)
   ↓
Service(80)
   ↓
Pod(80)
항목	의미
nodePort	외부 노출 포트
port	Service 포트
targetPort	실제 컨테이너 포트
🎙 대본

nodePort는 외부에서 들어오는 포트,
port는 Service의 포트,
targetPort는 컨테이너 포트입니다.

이 부분이 가장 헷갈리는 지점입니다.

6️⃣ Service는 몇 레이어인가?

Service는 L4 계층(TCP/UDP)에서 동작합니다.

HTTP 경로 기반 라우팅은 하지 못합니다.

그건 Ingress의 역할입니다.

Ingress Controller 예:

NGINX

Traefik

7️⃣ 실무 전체 구조
🖼 그림
Internet
   ↓
LoadBalancer
   ↓
Ingress
   ↓
ClusterIP
   ↓
Pods
🎙 대본

실제 운영 환경에서는
LoadBalancer + Ingress + ClusterIP 구조가 일반적입니다.

Service는 그 가운데에서 Pod를 안정적으로 묶는 핵심 요소입니다.

🎯 마무리 한 문장

Kubernetes Service는 Pod를 안정적으로 묶고, 고정된 네트워크 접근 지점을 제공하며, kube-proxy를 통해 L4 로드밸런싱을 수행하는 핵심 네트워크 객체입니다.

1️⃣ Ingress
📌 한 문장 정의

Ingress는 HTTP/HTTPS 트래픽을 경로(Path) 또는 도메인 기반으로 라우팅하는 L7 레벨 리소스입니다.

🖼 그림
Internet
   ↓
Ingress Controller
   ↓
Service A  (/api)
Service B  (/admin)
🎙 발표 대본

Service는 L4 수준입니다.
즉 TCP/UDP까지만 처리합니다.

하지만 실제 웹 서비스에서는
경로 기반 라우팅이 필요합니다.
예를 들어

example.com/api → backend
example.com/admin → admin

이 역할을 하는 것이 Ingress입니다.

📌 Ingress는 단독으로 동작하지 않는다

Ingress 리소스만 만들면 아무 일도 안 일어납니다.

반드시 Ingress Controller가 필요합니다.

대표적인 예:

NGINX

Traefik

📌 Service와의 관계
Ingress → Service → Pod

Ingress는 Service를 대상으로 라우팅합니다.
Pod를 직접 선택하지 않습니다.

2️⃣ CNI (Container Network Interface)
📌 한 문장 정의

CNI는 Pod 네트워크를 실제로 구성해주는 플러그인 시스템입니다.

🖼 그림
Node 1                    Node 2
[ Pod A 10.244.1.3 ]  ↔  [ Pod B 10.244.2.5 ]
        ↑                        ↑
        └──── CNI Network ───────┘
🎙 발표 대본

Kubernetes는 기본적으로
"모든 Pod는 서로 직접 통신 가능해야 한다"
라는 네트워크 모델을 가집니다.

그런데 Pod는 여러 노드에 흩어져 있습니다.

이걸 가능하게 해주는 게 CNI 플러그인입니다.

📌 대표 CNI

Calico

Flannel

Cilium

📌 Service와의 관계

Service가 Pod로 트래픽을 보내려면
Pod 간 통신이 가능해야 합니다.

그 통신을 실제로 구현하는 게 CNI입니다.

즉:

Service는 "논리적"
CNI는 "물리적 네트워크 구현"
3️⃣ CoreDNS
📌 한 문장 정의

CoreDNS는 Kubernetes 내부 DNS 서버입니다.

🖼 그림
Pod
  ↓  (fastapi-service 요청)
CoreDNS
  ↓
ClusterIP 반환
  ↓
Service
🎙 발표 대본

Kubernetes 내부에서는
Service 이름으로 접근합니다.

예를 들어
fastapi-service.default.svc.cluster.local

이 이름을 IP로 변환해주는 것이 CoreDNS입니다.

📌 DNS 구조

형식:

<service>.<namespace>.svc.cluster.local

예:

backend.default.svc.cluster.local
📌 Service와의 관계

CoreDNS는 Service 이름을
ClusterIP로 변환합니다.

즉:

Pod → CoreDNS → Service IP
🔥 세 개를 한 번에 정리
구성요소	역할	레이어
Ingress	HTTP 라우팅	L7
Service	TCP/UDP 로드밸런싱	L4
CNI	Pod 네트워크 구성	L3
CoreDNS	이름 → IP 변환	DNS
🎯 전체 흐름 (스터디에서 꼭 보여주기)
Internet
   ↓
LoadBalancer
   ↓
Ingress
   ↓
Service
   ↓
kube-proxy
   ↓
Pod
   ↓
CNI 네트워크

그리고 내부 호출은:

Pod → CoreDNS → Service → Pod

1️⃣ OSI 7계층 간단 정리
계층	이름	예시
L7	응용 계층	HTTP, HTTPS
L6	표현 계층	TLS
L5	세션 계층	세션 관리
L4	전송 계층	TCP, UDP
L3	네트워크 계층	IP
L2	데이터 링크	MAC
L1	물리 계층	케이블
🎯 Kubernetes에서 말하는 계층은?

우리가 Kubernetes 설명할 때:

L7 → HTTP 레벨 제어

L4 → TCP/UDP 포트 레벨 제어

L3 → IP 라우팅

DNS → 이름 해석 (응용 계층 기능)

이렇게 단순화해서 사용합니다.

🧠 예시로 이해하기
🔹 Ingress는 L7

왜냐하면:

URL 경로 기반 라우팅

Host 헤더 기반 분기

TLS 종료

예를 들어:

NGINX

Traefik

이런 Ingress Controller는 HTTP 헤더를 보고 판단합니다.
👉 이건 OSI L7 (Application Layer) 기능입니다.

🔹 Service는 L4

Service는:

TCP 포트

UDP 포트

기반으로 트래픽 전달합니다.

예:

10.96.0.10:80 → Pod:80

HTTP 경로는 신경 안 씁니다.

👉 그래서 L4라고 부릅니다.

🔹 CNI는 L3/L2

CNI는:

Pod IP 할당

노드 간 라우팅

VXLAN / BGP 등 구현

대표 예:

Calico

Cilium

이건 IP 레벨 처리이므로 L3 중심입니다.

🔹 CoreDNS는 어디인가?

DNS는 응용 계층 프로토콜 (L7) 입니다.

하지만 역할은:

이름 → IP 변환

즉, 네트워크 통신 전에 수행되는 해석 단계입니다.

🔥 전체를 OSI 관점으로 정리
L7  Ingress (HTTP 라우팅)
L7  CoreDNS (이름 해석)
L4  Service (TCP/UDP 분산)
L3  CNI (IP 네트워크 구성)
L2  Node 네트워크
L1  물리 서버
🎯 중요한 포인트

Kubernetes에서 “L4다”, “L7이다”라고 말할 때는
OSI 교과서 정확 구분이라기보다는 기능적 구분입니다.

즉,

HTTP 레벨 제어하면 L7

TCP 포트 레벨이면 L4

IP 레벨이면 L3

이 정도로 이해하면 충분합니다.
